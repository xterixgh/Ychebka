## 2.4 Тестирование программного продукта

Для проверки качества программного продукта применены два подхода: тестирование методом «чёрного ящика» и методом «белого ящика». Первый показывает, удобно ли работать с программой конечному пользователю; второй проверяет отдельные фрагменты кода с учётом внутренней логики.

### Ручные проверки (чёрный ящик)

«Тестирование методом чёрного ящика» — проверка приложения с позиции пользователя: ввод данных через интерфейс, сравнение результата на экране с ожидаемым, без анализа исходного кода.

Проверялись вход в систему и регистрация заказчика, создание и сопровождение заказа, приём заказа менеджером, перевод в производство со списанием со склада, сохранение планировки цеха директором. Для каждого сценария выполнялись типовые действия в интерфейсе Avalonia; при неверных данных система должна была выводить сообщение об ошибке, при корректных — сохранять изменения в PostgreSQL.

### Модульные тесты (белый ящик)

«Тестирование методом белого ящика» — анализ и проверка модулей с учётом алгоритмов, ветвлений и граничных значений. В проекте реализованы «модульные тесты» в отдельном решении «YchebnayaPractika.Tests» на фреймворке xUnit. Тестируемая логика вынесена в папку «Services» основного проекта: формирование номера заказа, проверка пароля при регистрации, определение доступных действий заказчика по статусу заказа.

| Класс тестов | Что проверяется |
|--------------|-----------------|
| PasswordValidatorTests | Допустимый пароль; отклонение короткого, без цифры, без заглавной, с запрещёнными символами |
| OrderNumberBuilderTests | Формат номера (12 символов), инкремент за день, сброс после 99-го заказа |
| CustomerOrderPermissionsTests | Доступность «Редактировать», «Удалить», «Отменить» для статусов заказа |

В тестах вызывается метод с заданными входными данными, результат сравнивается с ожидаемым через Assert (Рисунок 2.7).
