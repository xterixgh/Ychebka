## ВВЕДЕНИЕ

Предприятия, выпускающие конвейерное оборудование, ежедневно обрабатывают сведения о заказах, материалах, комплектующих, спецификациях, сотрудниках и планах цехов. При разрозненном учёте возрастает число ошибок и усложняется контроль выполнения заказов [1], [6].

Сопровождение заказа требует участия заказчика, менеджера, конструктора, мастера и директора. Каждой роли нужны свои данные в нужный момент. Обмен информацией без единой системы замедляет согласование сроков, стоимости и производственных операций.

Поэтому целесообразно использовать программу с общей базой данных и интерфейсом по ролям. Так проще хранить сведения, проверять ввод, выполнять расчёты и получать отчёты.

Целью работы является разработка десктопного приложения с базой данных для автоматизации учёта производства конвейеров и проверка работоспособности реализованных функций. Программа должна обеспечивать авторизацию, ведение заказов и справочников, спецификации изделий, расчёт ресурсов, планировку цехов и формирование текстовых отчётов.

Для достижения цели изучается предметная область, проектируется база данных в PostgreSQL и заполняется тестовыми данными. Разрабатывается интерфейс на C# с использованием Avalonia UI, доступ к данным выполняется через Entity Framework Core [2], [3], [4]. Реализуется разграничение прав для заказчика, менеджера, конструктора, мастера и директора. Проводится тестирование основных сценариев и оформляется отчёт.

Объектом исследования являются процессы учёта заказов и производства на предприятии. Предметом исследования — методы разработки программных модулей и интеграции приложения с СУБД PostgreSQL.

Практическая значимость работы состоит в централизации данных, сокращении ручного труда и согласованной работе всех участников процесса в одной информационной системе.

## ИНДИВИДУАЛЬНОЕ ЗАДАНИЕ

Индивидуальное задание предполагает разработку графического приложения на языке C# с использованием Avalonia UI и интеграцию с СУБД PostgreSQL. Программный продукт предназначен для автоматизации учёта на предприятии, выпускающем конвейерное оборудование.

Необходимо спроектировать базу данных в PostgreSQL и заполнить её тестовыми данными. В базе хранятся пользователи, заказы, материалы, комплектующие, изделия, спецификации, работники, планы цехов, сбои оборудования и контроль качества.

Приложение поддерживает пять ролей. Заказчик оформляет заказы, добавляет размеры и чертежи, редактирует заказ в статусе «Новый». Менеджер принимает и отклоняет заказы, ведёт статусы, считает ресурсы, строит диаграмму Ганта и формирует отчёт по складу.

Конструктор задаёт стоимость и плановую дату, переводит заказ на подтверждение. Мастер ведёт спецификации, регистрирует сбои и контроль качества. Директор управляет работниками, планировкой цехов и справочниками.

Разработка идёт в три сессии: справочники и работники; заказы, цеха, списание материалов; спецификации, расчёты, Ганта, отчёты по остаткам.

Номер заказа создаётся автоматически. При переходе в производство материалы списываются по спецификации. При нехватке на складе действие запрещается. Программа проверяет ввод и сообщает об ошибках.

Все созданные экраны должны быть связаны с уже реализованными разделами приложения. По итогам выполнения задания система должна обеспечивать полный цикл работы с заказом — от оформления заявки до закрытия — и соответствовать требованиям методических материалов по трём сессиям разработки.

## 1 ОБОСНОВАНИЕ ВЫБОРА СРЕДСТВ РАЗРАБОТКИ

Для реализации программного модуля по автоматизации учёта производства конвейеров необходимо выбрать язык программирования, среду разработки и систему управления базами данных. От этого зависит скорость создания приложения, удобство сопровождения кода и надёжность хранения данных. Ниже приведено обоснование выбранного стека технологий.

### 1.1 Язык программирования

В качестве основного языка разработки выбран C# — современный объектно-ориентированный язык платформы .NET. Он широко применяется при создании десктопных, веб- и серверных приложений и поддерживается обширной документацией и сообществом разработчиков [1].

C# сочетает строгую статическую типизацию с удобным синтаксисом. Поддерживаются классы, наследование, инкапсуляция и полиморфизм, что позволяет строить структурированное приложение с разделением интерфейса, логики работы с данными и моделей предметной области. Для учебного проекта с несколькими ролями пользователей и множеством экранов такой подход упрощает организацию кода.

Платформа .NET предоставляет богатую стандартную библиотеку классов: работа с файлами, коллекциями, датами, асинхронными операциями и сетевым взаимодействием. Кроссплатформенность современных версий .NET позволяет запускать приложение на разных операционных системах без переписывания основной логики.

Отдельным преимуществом является автоматическая сборка мусора. Разработчику не нужно вручную освобождать память, как в языках семейства C++, что снижает риск утечек при работе с формами, списками данных и загрузке файлов в базу.

Для доступа к PostgreSQL в проекте используется Entity Framework Core — ORM-фреймворк, который связывает таблицы базы данных с классами C#. Это ускоряет разработку CRUD-операций, фильтрации и загрузки связанных сущностей (заказы, материалы, спецификации) без написания большого объёма SQL-кода в приложении [4].

Выбор C# и .NET обоснован зрелостью технологии, удобством разработки десктопных приложений и хорошей интеграцией с выбранной СУБД через EF Core.

### 1.2 Среда разработки

В качестве среды разработки (IDE) выбран JetBrains Rider. Это кроссплатформенная IDE, ориентированная на проекты .NET, C# и связанные технологии [5].

Rider предоставляет интеллектуальное автодополнение кода, подсветку ошибок, быстрый переход к определениям классов и методов, рефакторинг и встроенную поддержку системы контроля версий Git. Эти возможности ускоряют написание и исправление кода при создании множества экранов Avalonia и классов доступа к данным.

Среда имеет встроенные средства для работы с базами данных: просмотр таблиц, выполнение запросов и анализ структуры PostgreSQL без переключения на отдельную программу. Это удобно при отладке заказов, справочников материалов и проверке результатов списания со склада.

Rider хорошо интегрируется с NuGet — менеджером пакетов .NET. Через него в проект подключены Avalonia UI, Entity Framework Core, провайдер Npgsql для PostgreSQL и другие зависимости.

IDE поддерживает визуальное редактирование XAML-разметки интерфейса и отладку приложения с точками останова. Это важно при проверке сценариев авторизации, смены статусов заказа и сохранения планировки цехов.

Выбор Rider обоснован специализацией на .NET-разработке, высокой производительностью и наличием инструментов для полного цикла создания приложения — от написания кода до работы с базой данных.

### 1.3 СУБД

В качестве системы управления базами данных выбран PostgreSQL — свободная реляционная СУБД с открытым исходным кодом. Она подходит для хранения структурированных данных учебного и прикладного характера [3].

PostgreSQL поддерживает стандарт SQL, внешние ключи, ограничения целостности, транзакции и индексы. Для проекта с заказами, справочниками, спецификациями и историей статусов это позволяет построить связную схему данных и избежать противоречивых записей.

СУБД надёжно работает с типами данных, необходимыми в приложении: числа, даты, текст, а также двоичные поля (bytea) для хранения изображений планов цехов и файлов чертежей. Поддержка identity-колонок упрощает создание первичных ключей в таблицах.

PostgreSQL хорошо сочетается с Entity Framework Core через провайдер Npgsql. Модели C# отображаются на таблицы базы, а запросы выполняются на уровне приложения. Для учебной практики также удобно выполнять SQL-скрипты создания таблиц и наполнения тестовыми данными в pgAdmin или через Rider.

Среди преимуществ PostgreSQL — стабильность, производительность на средних объёмах данных, бесплатное использование и широкое распространение в учебных и коммерческих проектах.

Выбор PostgreSQL обоснован соответствием требованиям реляционной модели данных, поддержкой бинарных файлов и удобной интеграцией с выбранным стеком C# и Entity Framework Core.

## 2 РАЗРАБОТКА ПРОГРАММНОГО ПРОДУКТА

В данном разделе описаны анализ предметной области, проектирование базы данных, реализация интерфейса и функционала приложения, тестирование, а также работа заказчика в системе.

### 2.1 Анализ предметной области

Предметной областью работы является деятельность предприятия, специализирующегося на производстве конвейерного и подъёмно-транспортного оборудования. Основной бизнес-процесс — выполнение заказа клиента: от получения заявки до изготовления изделия, проверки качества и передачи готовой продукции.

На предприятии одновременно обрабатывается информация разного характера. Это сведения о заказах и их статусах, данные о материалах и комплектующих на складах, технологические спецификации на изделия, информация о сотрудниках и выполняемых операциях, а также планы цехов с размещением оборудования и средств безопасности. При учёте без единой системы возникают задержки, ошибки при передаче данных между подразделениями и сложности при контроле сроков.

В процессе анализа выделены основные участники и их задачи.

Заказчик — физическое или юридическое лицо, размещающее заказ на изготовление конвейера. Ему необходимо оформить заявку, указать требования к изделию, при необходимости приложить чертежи и отслеживать статус выполнения. На раннем этапе заказчик может уточнять или отменить заявку.

Менеджер принимает входящие заказы, согласует условия с заказчиком, назначает ответственного и ведёт заказ по этапам. Менеджеру нужна информация о наличии материалов, сроках поставки, себестоимости и возможности запуска производства. Он также формирует отчёты по складским остаткам.

Конструктор участвует на этапе расчёта заказа: определяет стоимость изготовления и плановую дату завершения, после чего заказ передаётся на подтверждение заказчику.

Мастер отвечает за производственную часть: ведёт спецификации изделий (материалы, комплектующие, сборочные единицы, операции), сопровождает заказ в производстве, регистрирует сбои оборудования и проводит контроль качества готовой продукции.

Директор осуществляет общее управление: просматривает заказы и отчёты, ведёт учёт работников (мастеров и конструкторов), редактирует планировку цехов и имеет полный доступ к справочникам материалов и комплектующих.

Важным элементом предметной области является жизненный цикл заказа. Заказ проходит статусы: «Новый», «Составление спецификации», «Подтверждение», «Закупка», «Производство», «Контроль», «Готов», «Закрыт». Также предусмотрены статусы «Отменен» и «Отклонен». Каждое изменение статуса должно сохраняться в истории с датой и указанием пользователя, выполнившего действие.

Отдельно рассматриваются справочники. Материалы и комплектующие хранятся с указанием артикула, типа, единицы измерения, остатка на складе, поставщика и цены. Изделия описываются спецификацией, которая может включать вложенные сборочные единицы. При запуске производства потребность в ресурсах рассчитывается по спецификации с учётом количества в заказе.

При переходе заказа из статуса «Закупка» в «Производство» должно выполняться автоматическое списание материалов и комплектующих со склада. Если остатков недостаточно, система не должна разрешать смену статуса.

Директор ведёт планировку цехов: на схеме отмечаются оборудование, выходы, огнетушители и аптечки; координаты значков хранятся в базе данных. Проведённый анализ показал, что для описанных процессов целесообразна единая система на PostgreSQL с приложением на C# и Avalonia UI и разграничением прав для пяти ролей.

### 2.5 Интерфейс и функционал заказчика

Заказчик — пользователь, который оформляет заявки на изготовление конвейерного оборудования и следит за их выполнением. Для этой роли предусмотрены регистрация, вход и работа только со своими заказами; справочники склада, планировка цехов и учёт сотрудников недоступны.

Новый пользователь на экране авторизации (п. 2.3) нажимает «Регистрация», заполняет логин, пароль и ФИО. Система проверяет уникальность логина и требования к паролю. При успехе в базу добавляется учётная запись с ролью «Заказчик», выполняется возврат на экран входа.

Для работы заказчик вводит логин и пароль и нажимает «Войти». Открывается главное меню заказчика с кнопками «Заказы» и «Выход из сессии» (Рисунок 2.9).

Рисунок 2.9 — Главное меню заказчика

[ВСТАВИТЬ РИСУНОК 2.9]

По кнопке «Заказы» открывается экран списка личных заявок заказчика (Рисунок 2.10).

Рисунок 2.10 — Экран «Список заказов» заказчика

[ВСТАВИТЬ РИСУНОК 2.10]

Слева отображается перечень заказов с номером, датой и статусом; справа — блок «Детали заказа» по выбранной строке. Вверху доступны фильтр, кнопки «Добавить заказ» и «Обновить», внизу — действия над заказом и возврат «Назад» в главное меню.

Оформление новой заявки выполняется через «Добавить заказ»: указываются наименование, изделие из каталога, количество и описание, при необходимости добавляются размеры и файлы чертежей, после чего нажимается «Сохранить». Система присваивает номер заказа и статус «Новый».

Страница «Оформление заказа» состоит из нескольких логических блоков: общие параметры заявки, размеры готового изделия и список прикреплённых файлов. В верхней части пользователь заполняет наименование заказа и выбирает изделие из справочника. Ниже задаётся количество и текстовое описание, в котором фиксируются технические требования или дополнительные условия по изготовлению.

Блок «Размеры готового изделия» предназначен для поэтапного добавления параметров: сначала вводится название размера (например, длина, ширина, высота), затем числовое значение и единица измерения. Кнопка «Добавить размер» переносит запись в список, а «Удалить размер» позволяет убрать ошибочно добавленный параметр до сохранения заявки. Такой способ ввода снижает вероятность пропуска важных габаритов и делает карточку заказа более понятной для конструктора и мастера на следующих этапах обработки.

Отдельный блок отвечает за прикрепление файлов чертежей. Заказчик может добавить один или несколько файлов с локального компьютера, после чего имена файлов отображаются в списке на форме. Это позволяет сразу передать конструктору графические материалы, не отправляя их отдельно через сторонние каналы связи.

После нажатия «Сохранить» система выполняет проверку обязательных полей, формирует номер заказа по внутреннему правилу и записывает заявку в базу данных вместе с введёнными размерами и прикреплёнными файлами. При корректном сохранении пользователь возвращается к списку заказов, где новая запись отображается со статусом «Новый».

Для выбранного заказа доступны не все операции, а только разрешённые по статусу: в «Новый» — «Редактировать» и «Удалить»; в «Новый», «Составление спецификации» и «Подтверждение» — «Отменить». Кнопки приёма заказа и перевода в производство заказчику не отображаются.

Интерфейс заказчика выстроен как цепочка: регистрация → вход → меню → список заказов. Пользователь оформляет заявки, отслеживает статус в списке и деталях, уточняет данные на ранних этапах; дальнейшая обработка заказа выполняется другими ролями в той же базе данных.

Практика показала, что страница оформления заказа удобна для первичного сбора требований от заказчика и снижает количество уточнений на последующих этапах. За счёт поэтапного заполнения полей пользователь последовательно фиксирует обязательные и дополнительные параметры изделия, не пропуская ключевые сведения. Добавление размеров в отдельный список помогает структурировать технические характеристики и облегчает их проверку перед сохранением. Возможность приложить файлы чертежей на той же странице сокращает время передачи данных между заказчиком и конструктором. В результате карточка заказа сразу содержит достаточный объём информации для дальнейшего расчёта, согласования и запуска в работу.
