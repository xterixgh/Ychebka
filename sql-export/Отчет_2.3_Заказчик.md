## 2.3 Реализация интерфейса и функционала

Десктопное приложение построено на платформе Avalonia UI. Каждый экран оформлен отдельным пользовательским элементом (UserControl) в папке Pages проекта. Внешний вид задаётся разметкой XAML, обработка кнопок и обращение к базе данных — в связанном файле на C# (code-behind). Главное окно приложения подменяет содержимое рабочей области при переходе между экранами.

### Страница авторизации

Стартовым экраном служит форма авторизации класса LoginControl. При запуске в главное окно загружается именно этот элемент. Пользователь видит поля ввода логина и пароля, флажок «Запомнить меня» и три кнопки: «Войти», «Выход» и «Регистрация». Сообщения об ошибках выводятся красным текстом под кнопками (Рисунок 2.4).

Рисунок 2.4 — Окно авторизации

Разметка интерфейса описана в файле LoginControl.axaml. Элементы размещены в центре окна внутри карточки form-card. Для пароля у поля ввода задан символ маскировки PasswordChar. Обработчики нажатий кнопок привязаны к методам класса LoginControl (листинг 2.1).

«Листинг 2.1 — Фрагмент разметки страницы авторизации (LoginControl.axaml)»

```
<TextBlock Text="Логин"/>
<TextBox Name="LoginTextBox" Watermark="Введите логин"/>

<TextBlock Text="Пароль"/>
<TextBox Name="PasswordTextBox" PasswordChar="*" Watermark="Введите пароль"/>

<CheckBox Name="RememberMeCheckBox" Content="Запомнить меня"/>

<Button Content="Войти" Click="LoginButtonClick"/>
<Button Content="Выход" Click="ExitAppClick"/>
<Button Content="Регистрация" Click="RegisterButtonClick"/>

<TextBlock x:Name="MessageTextBlock" Foreground="..."/>
```

Логика входа реализована в методе LoginButtonClick файла LoginControl.axaml.cs. Сначала выполняется проверка: логин и пароль не должны быть пустыми. Затем выполняется асинхронный запрос к таблице UserAccount через Entity Framework Core: ищется запись, у которой совпадают логин и пароль, одновременно подгружается связанная роль пользователя (Include). Если учётная запись не найдена, поле пароля очищается и выводится текст «Неверный логин или пароль». При успешной проверке текущий пользователь сохраняется в общем объекте SessionData.CurrentUser, при отмеченном «Запомнить меня» логин и пароль записываются в локальный файл remember.txt, после чего вызывается переход на экран соответствующей роли (листинг 2.2).

«Листинг 2.2 — Проверка учётных данных и вход (LoginControl.axaml.cs)»

```
var user = await App.DbContext.UserAccounts
    .Include(u => u.IdUserRoleNavigation)
    .FirstOrDefaultAsync(u => u.Login == login && u.PasswordHash == password);

if (user == null)
{
    MessageTextBlock.Text = "Неверный логин или пароль.";
    return;
}

SessionData.CurrentUser = user;
OpenRoleScreen(user);
```

Выбор экрана после входа выполняется в методе OpenRoleScreen по имени роли из базы данных. Для заказчика открывается CustomerControl, для менеджера — ManagerControl, для директора — DirectorControl, для мастера — MasterControl, для конструктора — ConstructorControl. Таким образом одна форма авторизации обслуживает всех пользователей системы, а дальнейший интерфейс зависит только от роли (листинг 2.3).

«Листинг 2.3 — Переход на экран роли (LoginControl.axaml.cs)»

```
switch (user.IdUserRoleNavigation.RoleName)
{
    case "Заказчик":
        mainWindow.MainControl.Content = new CustomerControl();
        break;
    case "Менеджер":
        mainWindow.MainControl.Content = new ManagerControl();
        break;
    // ... Директор, Мастер, Конструктор
}
```

Кнопка «Регистрация» заменяет содержимое главного окна на RegisterControl — форму регистрации нового заказчика. Кнопка «Выход» закрывает приложение и удаляет файл remember.txt, если он был создан. При следующем запуске, если файл remember.txt существует и данные в нём ещё действительны, выполняется автоматический вход без повторного ввода пароля (метод AutoLogin).

Новый заказчик может создать учётную запись с экрана входа: «Регистрация» → поля логина, пароля и ФИО → «Зарегистрировать» (Рисунок 2.5).

### Главное меню заказчика

После успешного входа заказчик попадает на главный экран роли. На нём кратко указано назначение раздела и доступны две основные команды: переход к списку заказов и выход из сессии (возврат на экран авторизации). Другие разделы (справочники, работники, планировка цехов) для этой роли недоступны, что соответствует разграничению прав (Рисунок 2.6).

Рисунок 2.6 — Главное меню заказчика

### Список заказов и просмотр деталей

По кнопке «Заказы» открывается экран «Список заказов». Заказчик видит «только свои» заявки; чужие заказы в списке не отображаются. В верхней части экрана расположены фильтр и кнопки «Добавить заказ» и «Обновить».

Фильтр позволяет отобрать заказы по состоянию:

- «Все» — полный список;
- «Новые» — статусы «Новый», «Составление спецификации», «Подтверждение»;
- «Текущие» — «Закупка», «Производство», «Контроль»;
- «Выполненные» — «Готов», «Закрыт»;
- «Отклоненные» — «Отклонен», «Отменен».

Слева выводится перечень заказов в виде строк с номером, датой, статусом и наименованием; под списком показывается количество записей. При выборе строки справа в блоке «Детали заказа» отображаются номер, дата, наименование, статус, стоимость (если уже рассчитана конструктором), менеджер, плановая дата, изделие из каталога и текстовое описание. Если заказ отклонён, в карточке видна причина отклонения (Рисунок 2.7).

Рисунок 2.7 — Список заказов заказчика с деталями выбранного заказа

Для заказчика на этом экране доступны ограниченные действия по выбранному заказу:

- в статусе ««Новый»» — ««Редактировать»» и ««Удалить»» (полное удаление заявки);
- в статусах ««Новый»», ««Составление спецификации»» и ««Подтверждение»» — ««Отменить»» (перевод заказа в статус «Отменен»).

Кнопки смены этапов производства («Принять», «В закупку», «В производство» и т.п.) заказчику не показываются — ими управляют менеджер, конструктор и мастер. История смены статусов на этом экране заказчику также не выводится; при необходимости актуальный статус виден в списке и в блоке деталей.

### Оформление и редактирование заказа

Создание заявки выполняется по кнопке «Добавить заказ». Открывается форма Оформление заказа, в которой заказчик заполняет:

- «наименование заказа»;
- «изделие» — выбор из каталога продукции предприятия;
- «количество»;
- «описание изделия» (дополнительные требования).

Отдельно задаются «размеры готового изделия»: название замера, числовое значение и единица измерения; размеры можно добавить несколько и при необходимости удалить из списка.

В разделе ««Чертежи и схемы заказа»» заказчик прикрепляет один или несколько файлов (через диалог выбора файла). Имена файлов отображаются в списке; лишние вложения удаляются до сохранения.

После нажатия «Сохранить» система формирует «номер заказа» автоматически (по правилам предприятия: буквы из ФИО заказчика, дата и порядковый номер за день) и записывает заказ в базу со статусом «Новый». Размеры и файлы чертежей сохраняются вместе с заказом.

Редактирование доступно только для «собственного» заказа в статусе ««Новый»». В форме «Редактирование заказа» дополнительно показываются уже присвоенный номер и дата оформления; при наличии — рассчитанная стоимость и плановый срок (их вносит конструктор на следующих этапах). Заказчик может изменить наименование, изделие, количество, описание, размеры и вложения и сохранить изменения (Рисунок 2.8).

Рисунок 2.8 — Форма оформления заказа с размерами и чертежами

### Итог по роли заказчика

Таким образом, заказчик в приложении может зарегистрироваться и войти в систему, просматривать только свои заказы с фильтрацией по этапам, оформлять новые заявки с размерами и файлами, редактировать и удалять заказ на ранней стадии, а также отменить заявку до перехода в закупку. Остальные участники процесса (менеджер, конструктор, мастер, директор) работают с теми же данными заказа на следующих статусах — их интерфейс описывается отдельно.
